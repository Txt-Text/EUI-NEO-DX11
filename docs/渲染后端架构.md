# 渲染后端架构与流程

本文记录这个 fork 当前真实的渲染架构。目标不是继续维持多后端抽象叙事，而是把 Windows + Win32 + DX11 这条路径说明清楚，便于后续维护、调试和扩展。

## 当前定位

这个 fork 的渲染栈固定为：

- 窗口与消息循环：Win32
- GPU 设备与交换链：Direct3D 11
- 2D 图元绘制：Direct2D
- 文本布局与字体测量：DirectWrite
- UI 运行时：DSL Runtime

`core/render/render_backend.cpp` 当前固定解析到 DX11 后端，`core/window/window_types.h` 中的默认 `RenderApi` 也是 `Dx11`。

## 模块边界

```text
app / components / include/eui_neo.h
  |
core/app
  |
  +--> core/window        Win32 窗口、消息、光标、IME、剪贴板
  +--> core/input         后端无关输入状态与事件队列
  +--> core/dsl runtime   布局、状态、命中测试、动画、脏区
  +--> core/render        DX11 backend、图元、文本、图片
```

边界要求：

- `core/window` 负责 Win32 窗口语义，不负责绘制。
- `core/input` 负责输入归一化，不直接操作 GPU。
- `core/dsl runtime` 负责 UI 树、状态、脏区和渲染派发，不直接管理交换链。
- `core/render/dx11` 负责 D3D11 / D2D / DWrite 资源、绘制与呈现，不处理业务状态。

## 渲染职责划分

### Win32 窗口层

负责：

- 创建和销毁窗口
- 处理消息循环
- 提供鼠标、键盘、滚轮、IME 输入
- 处理剪贴板、系统光标、窗口图标
- 提供原生 `HWND`

不负责：

- 创建 UI 图元
- 管理 DSL 状态
- 执行 GPU 绘制命令

### Runtime

负责：

- 根据 DSL 构建 UI 树
- 执行 layout
- 维护交互状态、focus、文本输入和动画
- 计算 dirty rect
- 将 Rect / Text / Image / Polygon 派发到渲染后端

不负责：

- 直接创建 D3D11 设备
- 直接操作交换链
- 直接读取 Win32 消息

### DX11 Backend

负责：

- 创建 D3D11 device / context / swap chain
- 管理 Direct2D device context
- 管理 DirectWrite 文本绘制
- 维护 render cache 与 retained layer 资源
- 处理 clear、局部重绘、present、resize

不负责：

- 决定页面状态变化
- 处理业务事件
- 改写 DSL 结构

## 当前渲染流程

每帧的主路径可以概括为：

1. Win32 消息循环推进窗口消息。
2. Runtime 读取 pointer / keyboard / scroll 输入。
3. Runtime 更新命中测试、交互状态、动画和 dirty rect。
4. 如果需要重绘，Runtime 将图元派发到 DX11 backend。
5. DX11 backend 在 render cache 或 layer 上执行局部或整屏绘制。
6. DX11 backend 将结果提交到交换链并 present。

## 脏区与缓存

当前渲染优化围绕三层进行：

- dirty rect：只重绘变化区域
- render cache：复用离屏缓存，减少整屏重绘
- retained layer：缓存稳定静态子树

这些优化都建立在 Runtime 的保守矩形可见性判断之上。blur、依赖背景采样的内容、以及部分 layer 重建场景，会升级为 full paint 以保证正确性。

## 文本链路

文本系统当前基于 DirectWrite：

- 文本布局由 DirectWrite 负责
- 水平/垂直对齐通过 layout box 生效
- Runtime 传入文本 frame 的左上角、宽度和高度
- DX11 文本实现根据布局框绘制 icon 和普通文本

这意味着文本定位必须遵守标准盒模型：位置是布局框左上角，对齐由布局属性控制，而不是用“中心点”或“右下点”冒充绘制原点。

## 维护原则

后续所有渲染改动应遵守这些原则：

- 优先保持 Win32 + DX11 路径清晰可验证
- Runtime 和 backend 的职责边界必须稳定
- 文本、圆角、脏区、layer 等基础能力优先追求正确性，再做性能细化
- 文档必须反映当前 fork 的真实实现，而不是历史路线图


## 验证重点

涉及渲染架构或实现的修改，至少要验证：

- 普通 Rect / 圆角 / 边框 / 阴影是否正确
- Text / icon 的对齐、裁剪和 DPI 表现是否正确
- dirty rect 是否没有明显残影或漏刷
- blur 附近交互是否没有污染
- 窗口 resize、最小化、恢复、多窗口和 IME 是否正常
- Win32 覆盖遮挡下的命中测试是否正确
