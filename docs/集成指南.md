# 集成指南

本文面向这个 fork 的使用者。当前推荐的集成方式不是上游的跨平台 CMake 矩阵，而是围绕 Windows、Visual Studio、Win32 app entry 和 DX11 运行时来接入。

## 当前推荐路径

优先使用仓库里的 Visual Studio 工程，或参考同样的工程组织把源码接到你自己的 Windows 项目中。

当前事实边界：

- 平台：Windows
- 入口：`core/app/win32_app_main.cpp`
- 渲染：DX11 backend
- 语言标准：C++20

## 最小接入

最小应用只需要：

- 引入公共头文件 `eui_neo.h`
- 提供 `app::dslAppConfig()`
- 提供 `app::compose(eui::Ui&, const eui::Screen&)`
- 使用 Win32 app main 作为入口

最小 `app.cpp`：

```cpp
#include "eui_neo.h"

namespace app {

const DslAppConfig& dslAppConfig() {
    static const DslAppConfig config = DslAppConfig{}
        .title("My App")
        .pageId("my_app")
        .windowSize(960, 640)
        .fps(90.0f);
    return config;
}

void compose(eui::Ui& ui, const eui::Screen& screen) {
    ui.column("root")
        .size(screen.width, screen.height)
        .padding(32.0f)
        .content([&] {
            ui.text("title")
                .text("Hello EUI-NEO-DX11")
                .fontSize(28.0f)
                .build();
        })
        .build();
}

} // namespace app
```

## Visual Studio 工程方式

当前仓库内置：

- `EUI-NEO-DX11.slnx`
- `EUI-NEO-DX11.vcxproj`

推荐做法：

1. 打开 `EUI-NEO-DX11.slnx`
2. 选择 `Debug` 或 `Release`
3. 选择 `Win32` 或 `x64`
4. 直接构建和运行

如果你在自己的工程里接入，至少需要把这些部分纳入构建：

- `core/app/win32_app_main.cpp`
- `core/app/main_entry.cpp`
- `core/window/window_backend.cpp`
- `core/render/dx11/dx11_backend.cpp`
- `core/render/text_impl_bridge.cpp`
- 以及 `core/`、`components/`、`include/` 下所需头文件和源码

## 命令行构建

PowerShell 或 Developer Command Prompt 示例：

```powershell
msbuild .\EUI-NEO-DX11.vcxproj /p:Configuration=Release /p:Platform=x64
```

## 资源要求

运行时需要保留 `assets/` 目录中的字体、图标和图片资源。发布时应确保这些资源能被可执行文件找到。

当前 `Release` 构建产物不是自包含单 exe 发行物，这属于当前 fork 的预期设计；如果只分发可执行文件而不附带 `assets/`，字体、图标字体和部分图片资源会缺失。

默认字体和图标字体找不到时，文本系统会尝试回退到 Windows 系统字体；但 FontAwesome 这类图标字体仍建议随应用一起分发，避免 codepoint 对不上系统字体。


## 适合当前 fork 的接入原则

- 使用公共 facade 头文件写 UI，不直接操作底层图元
- 使用 `win32_app_main.cpp` 作为当前入口
- 使用当前 DX11 backend 作为渲染路径
- 以 Visual Studio 工程为主，不再把多后端构建矩阵作为推荐工作流
- 以当前仓库代码为准，不参考上游旧集成方式


## 嵌入式接入

如果你已经有自己的 Win32 程序，也可以把 DSL Runtime 和 DX11 backend 作为内部模块接入；但这类集成应由你自己负责：

- 主消息循环
- 窗口生命周期
- `HWND` 持有与消息分发
- 资源复制与打包
- 与现有渲染或工具链的边界

当宿主已经拥有自己的 D3D11 / D2D / DWrite 管线时，可以进一步使用实验性的外部上下文接管路径。`VA` 宿主的最小示例见 [`va_external_dx11_example.md`](va_external_dx11_example.md)，通用说明见 [`external_dx11_context.md`](external_dx11_context.md)。

除非确实需要和已有宿主程序深度耦合，否则优先沿用仓库现有 app entry。
